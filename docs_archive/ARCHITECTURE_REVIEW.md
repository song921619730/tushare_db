# tushare_db 架构评审记录

> 记录时间：2026-04-25
> 范围：tushare_db 数据仓库项目本身的设计问题与决策

---

## 项目概述

从零搭建 A 股数据仓库，服务本机 AI 与局域网 AI 的批量回测/分析。

| 维度 | 选择 |
|------|------|
| 平台 | Windows 11 + Docker Desktop + WSL2 |
| 数据库 | ClickHouse 24.8（容器，named volume 落 WSL2 ext4） |
| 调度 | APScheduler MemoryJobStore（不持久化） |
| 访问层 | ClickHouse 原生 HTTP/TCP + MCP Server SSE |
| 语言 | Python 3.11 |
| 接口覆盖 | 182 个 Tushare Pro 接口（227 个声明含 45 付费占位） |
| 数据回补 | 2020-01-01 起，五层顺序回补 |

---

## 已讨论并确认的问题

### 1. 接口数量：182 个全接，一个不砍

**结论**：全部保留。

tushare_db 的定位是 Tushare Pro 的本地化全量引擎，服务于 AI Agent 多维度深度调研。每个接口都是 AI 分析的一个维度，少一个接口，AI 的分析就少一块。

| 层级 | 接口数 | 内容 | AI 分析维度 |
|------|--------|------|------------|
| P0 | ~40 | 行情+资金流+龙虎榜+涨跌停+概念 | 主升浪、游资、异动 |
| P1 | ~60 | 财务+股东+融资融券+筹码 | 基本面、持仓结构 |
| P2 | ~80 | 宏观+债券+期货+基金+外汇 | 跨周期、跨资产 |

**回补阶段按 P0→P1→P2 排先后顺序，不是"做哪些不做哪些"，而是全部跑通。**

### 2. 四批次调度：保留，加延迟兜底

| 批次 | 时间 | 接口数 | 对应 Tushare 发布时段 |
|------|------|--------|---------------------|
| A 日线 | 17:00 | ~56 | 16:00-17:00 发布 |
| B 资金流 | 18:00 | ~19 | 17:00-18:00 发布 |
| C 龙虎榜 | 19:00 | ~43 | 18:00-19:00 发布 |
| D 财务/宏观 | 19:45 | ~38 | 19:00+ 发布 |
| Saturday 长尾 | 02:00 周六 | 4 | per_symbol_period |
| refresh_reference | 06:00 每日 | ~21 | 参考表 |
| weekly_reconcile | 02:00 周日 | — | 补洞扫描 |
| verify_row_counts | 03:00 每日 | — | 行数核对 |

**增加延迟兜底**：某批次跑完发现数据仍是 T-2 的（说明 Tushare 发布延迟），记录 WARN 并安排 1 小时后再试。

### 3. MCP Server 工具

| 工具 | 用途 |
|------|------|
| `query_sql` | 通用 SQL 查询（强制 SELECT/WITH/SHOW/DESCRIBE 前缀） |
| `describe_table` | 查看表结构 |
| `list_interfaces` | 列出已接入接口 |
| `get_ohlcv` | OHLCV 行情查询（含复权逻辑） |
| `trade_calendar` | 交易日历查询 |

语义化工具保留，AI 作为 CLI 使用者（Claude Code / Hermes Agent），直接调用比手写 SQL 更不易出错。

`tavily_search` 移除（已集成到其他自带 MCP 工具中）。

### 4. 外部数据源

| Agent 角色 | 数据源 | 覆盖缺口 |
|-----------|--------|---------|
| 期货专家 | 金十数据 API | 商品现货价格、库存、高炉开工率、基差 |
| 财经专家 | tushare_db (CH) | A 股核心数据 |

**对 tushare_db 的影响**：
- 不扩展到全球商品和新闻数据
- 金十数据由 Agent 侧独立管理（不接入 tushare_db）

### 5. 脏数据：现有机制已覆盖大部分

| 场景 | 现有方案 | 状态 |
|------|---------|------|
| 异常值（价格=0） | 样本推断自动处理 NULL | ✅ |
| 停牌日无行 | 预期行为，不视为脏数据 | ✅ |
| 数据延迟 | 四批次调度 + 延迟兜底 | ✅ |
| Tushare 事后修正 | ReplacingMergeTree 天然处理 | ✅ |
| 列集静默变化 | schema_evolver ADD COLUMN + WARN | ✅ |
| 接口漏发（今天行数=0） | **需在 `verify_row_counts` 中加 WARN** | ⚠️ 待加 |

## 待讨论的问题

### 1. Windows + WSL2 + ClickHouse 运维

| 风险 | 现状 |
|------|------|
| named volume 备份/恢复困难 | 数据卷损坏需重跑 48-60h |
| Windows 挂载性能损耗 | 日志走 Windows 路径，有损耗 |
| 宿主机内存限制 | host 16GB RAM，CH 容器 12GB limit |

**待确认**：用户是否接受"数据卷损坏 → 重跑"的代价？如果不可接受，评估 PostgreSQL 替代。

### 2. Hermes Agent 定位

Hermes 定位为"策略执行者"，读策略 → 执行 → 交易。

**矛盾点**：
- 如果只是读 JSON 然后调券商 API，就是个定时脚本
- 如果要做盘中动态决策，需要实时数据——但 tushare_db 是盘后更新
- 策略文件格式太理想化：复杂条件无法用简单 JSON 表达

**待确认**：Hermes 接实时数据源（MiniQMT），还是降级为"信号监控器"？
