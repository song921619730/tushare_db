## Bug #1: `_normalize_items` 日期解析不完整

**影响范围**：所有接口的 date 类型列
**严重程度**：HIGH

### 根因
worker.py 的 `_normalize_items()` 函数只处理 8 位日期字符串（YYYYMMDD），但 Tushare API 实际返回多种格式：
- `'2004'`（年份）
- `'201301'`（年月）
- `'201106'`（年月）
- `'20200101'`（完整日期）
- `'2020-01-01'`（带分隔符）
- `'2020/01/01'`（斜杠分隔）

不支持的格式导致 `TypeError: unsupported operand type(s) for -: 'str' and 'datetime.date'`

### 修复方案
**文件**：`src/tushare_db/runner/worker.py`

添加 `_parse_date_string()` 函数，支持多种日期格式：

```python
def _parse_date_string(val: str) -> datetime.date:
    val = val.strip()
    if not val:
        raise ValueError("empty date string")
    if len(val) == 4 and val.isdigit():
        return datetime(int(val), 1, 1).date()
    elif len(val) == 6 and val.isdigit():
        return datetime.strptime(val, "%Y%m").date()
    elif len(val) == 8 and val.isdigit():
        return datetime.strptime(val, "%Y%m%d").date()
    elif len(val) >= 10:
        clean = val[:10].replace("-", "").replace("/", "")
        if len(clean) >= 8 and clean[:8].isdigit():
            return datetime.strptime(clean[:8], "%Y%m%d").date()
    raise ValueError(f"unrecognized date format: {val}")
```

## Bug #2: `_normalize_items` 1970 年前日期溢出

**影响范围**：stock_company (setup_date), etf_index (pub_date), index_basic (base_date)
**严重程度**：CRITICAL

### 根因
ClickHouse Date 类型以 UInt16 存储 1970-01-01 以来的天数（范围 0–65535，上限约 2149 年）。中国股市存在 1970 年前的日期：
- 招商局：1896-05-26
- 多家公司：1955, 1958, 1966 等

这些日期转为负的天数偏移量，ClickHouse 无法存储，抛出 `struct.error: ushort format requires 0 <= number <= 65535`

### 修复方案
**文件**：`src/tushare_db/runner/worker.py`

在日期归一化后增加 step 1c，将 1970 年前的日期截断：

```python
# 1c. Cap dates before ClickHouse epoch (1970-01-01)
for idx in all_date_indices:
    if idx >= len(row) or row[idx] is None:
        continue
    val = row[idx]
    if isinstance(val, datetime):
        if val.year < 1970:
            ch_type = column_types.get(fields[idx], "")
            if ch_type.startswith("Nullable("):
                row[idx] = None
            else:
                row[idx] = datetime(1970, 1, 1)
    elif isinstance(val, date) and not isinstance(val, datetime):
        if val.year < 1970:
            ch_type = column_types.get(fields[idx], "")
            if ch_type.startswith("Nullable("):
                row[idx] = None
            else:
                row[idx] = _EPOCH_DATE
```

## Bug #3: `_normalize_items` 空字符串未处理

**影响范围**：ccass_hold 等有空字符串的接口
**严重程度**：HIGH

### 根因
Tushare API 对非 Nullable 列可能返回空字符串 `""`。`_normalize_items` 未将空字符串转为 None 再填充默认值，导致 ClickHouse 报错 `DataError: Unable to create Python array for source column`

### 修复方案
**文件**：`src/tushare_db/runner/worker.py`

在 step 3 中增加空字符串处理：

```python
# 3. Fill None/empty defaults for non-nullable columns
for idx, field_name in enumerate(fields):
    ...
    # Treat empty strings and whitespace as None for typed columns
    if isinstance(val, str) and not val.strip():
        row[idx] = None
        val = None
    if val is not None:
        continue
    # Unwrap LowCardinality to determine base type
    base_type = ch_type
    if base_type.startswith("LowCardinality("):
        base_type = base_type[len("LowCardinality("):-1]
    # Fill sensible defaults
    if base_type == "String":
        row[idx] = ""
    elif base_type.startswith(("Int", "Float", "UInt", "Decimal")):
        row[idx] = 0
    elif base_type == "Date":
        row[idx] = _EPOCH_DATE
```

## Bug #4: `_normalize_items` DateTime 列被 date 名称启发式误处理

**影响范围**：stk_nineturn（802 个单元全部失败）
**严重程度**：HIGH

### 根因
stk_nineturn 的 `trade_date` 列类型是 `DateTime`，但字段名含 "date"，被名称启发式规则 `"date" in field.lower()` 捕获到 `all_date_indices`。

step 1a 将其转为 `datetime.date` 对象（只有日期，无时间）。step 1b 只处理 `str` 类型，跳过已转换的 `date` 对象。结果 `date` 对象被传入 DateTime 列，clickhouse_connect 调用 `.timestamp()` 时报错 `'datetime.date' object has no attribute 'timestamp'`

### 修复方案
**文件**：`src/tushare_db/runner/worker.py`

两处修复：

**1. 排除 DateTime 列的名称启发式：**
```python
# Before:
all_date_indices = date_type_indices | name_date_indices

# After:
all_date_indices = date_type_indices | (name_date_indices - datetime_type_indices)
```

**2. step 1b 增加 date→datetime 提升：**
```python
# 1b. DateTime normalization
for idx in datetime_type_indices:
    if idx >= len(row) or row[idx] is None:
        continue
    val = row[idx]
    # Convert date → datetime (name heuristic edge case)
    if isinstance(val, date) and not isinstance(val, datetime):
        row[idx] = datetime(val.year, val.month, val.day)
        continue
    if not isinstance(val, str):
        continue
    ...
```

## Bug #6: `ccass_hold` 列类型推断错误

**影响范围**：ccass_hold（1276 个单元全部失败）
**严重程度**：CRITICAL

### 根因
`shareholding` 字段存储的是持仓数量（如 `'656439907'`、`'1173194432'`），但 schema 推断将其识别为 `Date` 类型。API 返回的数字字符串无法转换为有效日期，导致 ClickHouse 序列化失败：`struct.error: ushort format requires 0 <= number <= 65535`

### 修复方案

**1. ALTER TABLE 修改列类型：**
```sql
TRUNCATE TABLE tushare.tushare_ccass_hold;
ALTER TABLE tushare.tushare_ccass_hold MODIFY COLUMN shareholding String;
```

**2. 配置文件增加 schema_overrides**（`config/interfaces/stock_special.yaml`）：
```yaml
name: ccass_hold
...
schema_overrides:
  shareholding: String   # was incorrectly inferred as Date
```

周期: 2020-01-01 至今 (~25 个 reporting periods)
预估: 5511 × 25 = ~137,750 单元

由于单元量巨大，建议：
1. 分批次执行 plan_units（每次一个季度）
2. 使用 4 个 special workers（避免过载）
3. 在周末或低 API 使用时段运行

### 问题 B: fund_sales_ratio / fund_sales_vol —— 5 次/分钟硬限制

**问题描述**：

| 接口 | 配置文件 | 策略 | 当前状态 | RPM 限制 |
|------|----------|------|----------|-----------|
| fund_sales_ratio | fund.yaml:119 | date_loop | 有 failed/running 残留 | **5 次/分钟** |
| fund_sales_vol | fund.yaml:136 | date_loop | 有 failed/running 残留 | **5 次/分钟** |

Tushare 对这两个基金销售接口实施了 **5 次/分钟** 的硬限制（区别于常规 500/min 和 special 285/min）。

此前以 normal 频率（12 workers, 475 RPM）回填导致大量频率超限错误。

**解决思路**：

1. **清理残留状态**：将 running/failed 记录重置为 pending
   ```sql
   ALTER TABLE _meta.sync_state DELETE WHERE interface_name IN ('fund_sales_ratio', 'fund_sales_vol') AND status IN ('running', 'failed');
   ```

2. **创建低速回填脚本**（`retry_fund_slow.py`）：
   ```python
   # 核心参数
   workers = 1           # 单 worker，避免并发抢频率
   delay_per_call = 15.0  # 每次 API 调用间隔 15 秒（60/5 = 12s 理论最小，留 3s 余量）
   max_retries = 2

   # 执行时间估算
   # fund_sales_ratio: date_loop, start=20200101, ~1531 个交易日
   # 1531 × 15s = 22,965s ≈ 6.4 小时
   # fund_sales_vol: date_loop, start=20210101, ~1273 个交易日
   # 1273 × 15s = 19,095s ≈ 5.3 小时
   ```

3. **执行策略**：
   - 两个接口串行执行（避免互相争抢 5/min 配额）
   - 不使用 execute_batch 的默认并发，使用 `executor.py` 中的 1 个 special worker
   - 或者直接写一个简单的顺序循环，每次 `time.sleep(15)` 后调用 API

4. **替代方案（如果时间不可接受）**：
   - 只回填近 1 年数据（而非从 2020 年开始）
   - 修改 `start_date` 为 `20240401`，将单元量从 ~1531 降到 ~250，时间从 6 小时降到 1 小时

### 问题 C: 16 个 Python 进程残留

**问题描述**：
- 环境中有 16 个 python.exe 进程持续运行
- 可能来源：旧批处理任务、调度器实例、MCP 服务器等
- 影响：所有进程共享 Tushare API 频率配额，导致当前回填任务受干扰（dividend 重试时仍出现频率超限）

**解决思路**：

1. **识别进程来源**：
   ```bash
   # 查看每个进程的启动命令和运行时间
   tasklist /FI "IMAGENAME eq python.exe" /V /FO CSV
   
   # 精确查看命令行参数（PowerShell）
   powershell "Get-WmiObject Win32_Process -Filter \"name='python.exe'\" | Select-Object ProcessId,CommandLine,CreationDate | Format-Table -AutoSize"
   ```

2. **分类处理**：
   | 进程类型 | 处理方式 |
   |----------|----------|
   | 旧批处理任务（retry_v2.py 等） | 立即杀掉 — 任务已完成 |
   | 调度器实例（scheduler-run） | 确认是否有必要留多个，保留 1 个 |
   | MCP 服务器（mcp-serve） | 保留 1 个 |
   | 无法识别的 | 杀掉后观察 |

3. **建议保留的进程**：
   - 1 个调度器（负责增量更新）
   - 1 个 MCP 服务器（SSE 7800 端口）
   - Docker 容器中的进程

4. **清理后验证**：
   ```bash
   # 确认只保留必要的进程
   tasklist /FI "IMAGENAME eq python.exe" /V
   ```

5. **预防措施**：
   - 批处理任务脚本加上 `atexit` 清理逻辑或在完成后明确退出
   - 使用 `tushare-db status` 检查是否有 running 状态的单元（说明有残留任务）
   - 定期 `tasklist` 检查，防止进程堆积
